# 推荐算法说明文档

> 对应源码文件：`src/main/java/com/algorithm/recommend/RecommendAlgorithmFactory.java`

## 一、总体概述

`RecommendAlgorithmFactory` 是本项目的**推荐系统核心引擎**，采用工厂模式，提供 3 种推荐算法 + 模型管理能力。外部只需调用几个静态方法就能获得推荐结果，主要应用于**技能市场**、**学习资料**和**用户推荐**的个性化推荐场景。

---

## 二、整体架构

```
RecommendAlgorithmFactory（工厂入口）
│
├── UserBehavior              ← 数据载体：记录「谁 对 什么物品 打了多少分」
│
├── ModelStorage              ← 模型仓库：负责模型的保存/加载/缓存
│
├── UserItemMatrixRecommender ← 算法1：用户协同过滤（用协）
│
├── ItemUserMatrixRecommender ← 算法2：物品协同过滤（物协）
│
├── AlgorithmTrainBuilder     ← 训练器：训练机器学习模型（随机森林/神经网络）
│
└── ModelLoader               ← 混合推荐器：加载训练好的模型 + 协同过滤补充
```

---

## 三、各组件详解

### 1. UserBehavior — 用户行为数据载体

所有推荐算法的输入数据都是一组 `UserBehavior` 列表，每条记录包含三个字段：

| 字段 | 类型 | 含义 |
|------|------|------|
| `userId` | Long | 用户ID |
| `itemId` | Long | 物品ID（技能需求ID、学习资料ID等） |
| `score` | Double | 用户对该物品的评分（默认1.0） |

示例：

- 用户A 收藏了技能需求X → `UserBehavior(A, X, 1.0)`
- 用户B 收藏了学习资料Y → `UserBehavior(B, Y, 1.0)`

在本项目中，行为数据来源于 `storeup` 表（收藏 `type=1`、点赞 `type=21`），隐式评分统一为 **1.0**。

---

### 2. ModelStorage — 模型仓库

负责把训练好的模型**保存到磁盘**和**从磁盘加载**，并用内存缓存加速。

- **存储路径**：`algorithm/recommend/models/xxx.model`
- **核心方法**：
  - `saveModel()` → Java 序列化写文件 + 放入内存缓存
  - `loadModel()` → 先查缓存，没有再从文件反序列化
  - `hasModel()` → 检查模型是否存在
  - `clearCache()` → 清除内存缓存

---

### 3. UserItemMatrixRecommender — 用户协同过滤（用协）

**核心思想：「和你相似的用户喜欢什么，就给你推荐什么」**

工作流程：

```
第1步：构建「用户-物品」评分矩阵
         物品1  物品2  物品3  物品4
用户A     5      3      -      -
用户B     4      -      2      -
用户C     5      3      -      4

第2步：要给用户A推荐 → 找和A最像的用户
  计算A和B的余弦相似度
  计算A和C的余弦相似度
  → 发现C和A最像（都给物品1打了5分、物品2打了3分）

第3步：C喜欢但A没看过的物品 → 推荐物品4给A
  推荐分 = C对物品4的评分 × A和C的相似度
```

**余弦相似度**是数学核心，用来衡量两个用户的口味有多接近。把每个用户的评分看作一个向量，计算两个向量夹角的余弦值：

- 余弦值 = 1 → 口味完全一样
- 余弦值 = 0 → 完全没有关联

---

### 4. ItemUserMatrixRecommender — 物品协同过滤（物协）

**核心思想：「和你喜欢的东西相似的物品，推荐给你」**

和用户协同的区别是矩阵**转置**了：

```
第1步：构建「物品-用户」评分矩阵（物品为行，用户为列）
         用户A  用户B  用户C
物品1      5      4      5
物品2      3      -      3
物品3      -      2      -
物品4      -      -      4

第2步：用户A喜欢物品1 → 找和物品1最相似的物品
  物品1和物品2的相似度 = ?（看哪些用户都给它们打过分）
  物品1和物品4的相似度 = ?
  → 发现物品4和物品1很像（用户C都给了高分）

第3步：推荐物品4给用户A
  推荐分 = A对物品1的评分 × 物品1和物品4的相似度
```

**用协 vs 物协的区别**：

- 用协：适合用户少、物品多的场景，推荐更有探索性
- 物协：适合物品少、用户多的场景，推荐更稳定、可解释性更强

---

### 5. AlgorithmTrainBuilder — 机器学习模型训练器

这是**最复杂的部分**，使用 Weka 机器学习库训练推荐模型。支持两种算法：

| 算法 | Weka 实现类 | 配置参数 | 特点 |
|------|-------------|----------|------|
| **随机森林** | `RandomForest` | 100棵决策树，深度不限 | 多棵决策树组合投票，稳定可靠 |
| **NCF深度学习** | `MultilayerPerceptron` | 学习率0.3，动量0.2，500轮训练，隐藏层(50,25) | 两层隐藏层的神经网络，能捕捉非线性关系 |

> 本项目技能推荐的 `trainRecommend` 接口实际调用的是 **NCF 深度学习** 算法进行模型训练。

**训练流程**：

```
输入3类数据：
  ① 用户数据（性别、年龄）
  ② 物品数据（需求标题、技能分类）
  ③ 交互记录（用户对技能的收藏行为，隐式评分 1.0）
       ↓
第1步：提取特征 → 自动从用户和物品数据中识别所有字段
第2步：构建 Weka 训练数据集 → 每条交互记录变成一行特征向量
  - 数值型字段（年龄等）直接作为数值属性
  - 文本型字段（分类、性别等）作为分类属性，自动收集所有可能取值
第3步：训练模型 → MultilayerPerceptron（NCF深度学习）
第4步：保存模型文件（jinengxuqiu.model）+ 元数据（jinengxuqiu_metadata.ser）
```

**元数据**保存了训练时的所有辅助信息：

- `allItemIds`：所有物品ID列表
- `userFeatures`：每个用户的特征数据
- `itemFeatures`：每个物品的特征数据
- `featureNames`：特征字段名列表
- `userInteractedItems`：每个用户交互过的物品集合

---

### 6. ModelLoader — 混合推荐器

加载训练好的模型，做**混合推荐**（模型预测 + 协同过滤兜底）：

```
给用户A推荐10个物品：
  ↓
第1步：用训练好的机器学习模型预测
  对每个候选物品 → 输入(用户A特征 + 物品特征) → 模型输出预测分数
  按分数排序 → 得到模型推荐列表
  ↓
第2步：模型推荐够10个了？
  够了 → 直接返回
  不够 → 用协同过滤补充（默认用「用协」，也可以选「物协」）
  ↓
第3步：合并去重 → 返回最终推荐列表
```

这样做的好处是：模型能学到深层特征关系，协同过滤能兜底保证推荐数量。

## 四、外部调用方式

文件提供了 7 个静态工厂方法，外部使用非常简单：

```java
// 方式1：直接用「用户协同过滤」推荐
List<UserBehavior> behaviors = ...; // 从 storeup 表查出用户行为
var recommender = RecommendAlgorithmFactory.buildUserItemMatrix(behaviors);
List<Long> result = recommender.recommendItems(userId, 10);

// 方式2：直接用「物品协同过滤」推荐
var recommender = RecommendAlgorithmFactory.buildItemUserMatrix(behaviors);
List<Long> result = recommender.recommendItems(userId, 10);

// 方式3：训练机器学习模型
var trainer = RecommendAlgorithmFactory.buildAlgorithm("NCF深度学习", "jinengxuqiu");
trainer.train(userDataList, itemDataList, interactionList);

// 方式4：加载模型做混合推荐
var loader = RecommendAlgorithmFactory.loadModel("jinengxuqiu.model");
List<Long> result = loader.recommendItems(userId, 10);

// 方式5：检查模型是否已训练
boolean exists = RecommendAlgorithmFactory.hasModel("jinengxuqiu.model");

// 方式6：列出所有已训练模型
List<String> models = RecommendAlgorithmFactory.listModels();

// 方式7：清除缓存
RecommendAlgorithmFactory.clearCache();
```

| 方法 | 返回类型 | 说明 |
|------|----------|------|
| `buildUserItemMatrix(list)` | `UserItemMatrixRecommender` | 构建用户协同过滤推荐器 |
| `buildItemUserMatrix(list)` | `ItemUserMatrixRecommender` | 构建物品协同过滤推荐器 |
| `buildAlgorithm(type, name)` | `AlgorithmTrainBuilder` | 创建模型训练器 |
| `loadModel(name)` | `ModelLoader` | 加载已训练模型的混合推荐器 |
| `hasModel(name)` | `boolean` | 检查指定模型是否存在（缓存或文件） |
| `listModels()` | `List<String>` | 列出所有已训练的模型文件 |
| `clearCache()` | `void` | 清除用户矩阵、物品矩阵和模型缓存 |

## 五、在本项目中的应用场景

本推荐引擎主要用在**技能市场**、**学习资料**和**用户推荐**三个场景中：

| 场景 | 数据来源 | 调用入口 | 推荐策略 |
|------|----------|----------|----------|
| 技能推荐 | 用户需求向量 + 收藏/点赞行为 | `JinengxuqiuController` 的 `autoSort2` | 需求画像 → NCF模型 → User-CF → 热度兜底 |
| 技能模型训练 | 用户特征 + 物品特征 + 收藏行为 | `JinengxuqiuController` 的 `trainRecommend` | NCF 深度学习（MLP） |
| 学习资料推荐 | 用户需求向量 + 收藏行为 | `XuexiziliaoController` 的 `autoSort2` | 需求画像 → User-CF → 热度兜底 |
| 用户推荐 | 技能发布关系 + 需求相似度 + 信誉值 | `YonghuController` 的 `/recommend` | 技能提供者匹配 + 需求相似度 + 信誉加权 |

### 技能推荐流程（autoSort2）

推荐分为两个阶段：

**第一阶段 — 需求画像推荐**

1. 从所有"需求"类型记录中，统计每个用户在各技能分类上的发布数量，构建稀疏需求向量
2. 计算当前用户与其他用户的余弦相似度，取 Top-20 相似用户
3. 收集相似用户的收藏（权重 `sim × 2.0`）和点赞（权重 `sim × 1.0`）行为
4. 对需求分类内的热门技能补充 `+0.5` 得分

**第二阶段 — 模型 + CF 补充（结果不足时触发）**

5. 从所有用户的收藏和点赞行为构建用户-物品矩阵
6. 加载 `jinengxuqiu.model` 进行 NCF 模型预测
7. User-Based 协同过滤补充
8. 热门技能兜底（按点击量排序）

### 学习资料推荐流程（autoSort2）

与技能推荐结构类似，但有以下差异：

| 对比维度 | 技能推荐 | 资料推荐 |
|----------|----------|----------|
| 行为类型 | 收藏(×2) + 点赞(×1) | 仅收藏(×2) |
| MLP 模型 | 加载 `jinengxuqiu.model` 预测 | 不使用 MLP 模型 |
| 补充策略 | NCF → User-Based CF → 热门 | User-Based CF → 热门 |
| 兴趣来源 | 需求发布的技能分类 | 同左（跨领域复用） |

### 用户推荐流程（/recommend）

用于向当前用户推荐可能感兴趣的其他用户，默认推荐 6 位：

1. 从当前用户的需求向量中提取关注的技能分类
2. 查找这些分类下发布了"技能"类型内容且已审核通过的用户，基础分 `3.0`
3. 计算需求向量的余弦相似度，加权 `sim × 2.0`
4. 加入信誉值因子 `信誉值 / 100`
5. 对需求相似度 > 0.3 的用户额外加权
6. 按综合得分排序，不足时按信誉值降序兜底

## 六、答辩讲解参考

### 问题1：推荐功能是怎么做的？

系统实现了一个推荐算法模块，用来给用户个性化推荐技能、学习资料和相似用户。核心思路是：系统会记录用户的浏览、点赞、收藏这些行为，然后基于这些行为数据，用**协同过滤算法**计算用户之间的相似度，把相似用户喜欢的内容推荐给当前用户。

另外还结合了**机器学习模型**——用 Weka 的 MultilayerPerceptron（多层感知器）来做预测，把用户特征（比如性别、年龄）和物品特征（比如标题、分类）作为输入，预测用户对未看过的物品的感兴趣程度。最后把模型推荐和协同过滤的结果**混合**起来，保证推荐的质量和数量。

### 问题2：协同过滤具体怎么实现的？

实现了两种协同过滤：**用户协同过滤**和**物品协同过滤**。

以用户协同过滤为例，先构建一个**用户-物品评分矩阵**，行是用户，列是物品，值是评分。然后用**余弦相似度**来衡量两个用户的相似程度——把每个用户的评分看成一个向量，算两个向量夹角的余弦值，越接近1说明两个用户口味越像。

举个例子：用户A和用户C都给"Python编程"打了5分、给"吉他教学"打了3分，那他们的相似度就很高。如果C还收藏了"摄影技巧"但A没看过，系统就会把"摄影技巧"推荐给A，推荐分数等于C的评分乘以相似度。

物品协同过滤反过来，是看**物品之间的相似度**——如果"Python编程"和"数据分析"经常被同一批用户点赞，那喜欢Python编程的用户很可能也喜欢数据分析。

### 问题3：为什么用混合推荐？

因为单一算法都有局限性。协同过滤有**冷启动问题**——新用户没有行为数据的时候推荐不了东西；而机器学习模型虽然能利用用户和物品的特征来预测，但有时候候选物品太少，推荐数量不够。

所以采用了**混合策略**：优先用需求画像匹配，再用机器学习模型预测评分，如果数量不够，就用协同过滤来补充，最后热门内容兜底。这样既保证了推荐的准确性，又保证了推荐的数量。

### 问题4：用了什么机器学习算法？

本项目的 `trainRecommend` 接口实际训练的是 **NCF（Neural Collaborative Filtering）**，基于 Weka 的 `MultilayerPerceptron` 实现：

- 两个隐藏层，分别 50 和 25 个神经元
- 学习率 0.3，动量 0.2
- 训练 500 轮
- 输入特征：用户的性别和年龄 + 物品的标题和分类
- 训练数据：用户对技能的收藏行为（隐式评分 1.0）

它的优点是能捕捉用户和物品特征之间的**非线性关系**，比如"年轻女性用户"和"手工DIY类技能"之间的潜在关联。

算法框架层面同时支持**随机森林**（100棵决策树集成学习），可通过 `buildAlgorithm("随机森林", ...)` 切换。

### 问题5：余弦相似度的公式是什么？

$$\text{similarity}(A, B) = \frac{\sum_{k \in K} A_k \cdot B_k}{\sqrt{\sum_{i} A_i^2} \cdot \sqrt{\sum_{j} B_j^2}}$$

其中 $K$ 为两个用户共同涉及的特征维度集合，分子是共同维度上的评分乘积之和，分母是两个用户各自评分向量的 L2 范数相乘。结果在 $[0, 1]$ 之间，越大越相似。

代码实现：

```java
private double needCosineSimilarity(Map<String, Double> vec1, Map<String, Double> vec2) {
    Set<String> commonKeys = new HashSet<>(vec1.keySet());
    commonKeys.retainAll(vec2.keySet());
    if (commonKeys.isEmpty()) return 0.0;
    double dotProduct = 0.0;
    for (String key : commonKeys) {
        dotProduct += vec1.get(key) * vec2.get(key);
    }
    double norm1 = Math.sqrt(vec1.values().stream().mapToDouble(v -> v * v).sum());
    double norm2 = Math.sqrt(vec2.values().stream().mapToDouble(v -> v * v).sum());
    if (norm1 == 0 || norm2 == 0) return 0.0;
    return dotProduct / (norm1 * norm2);
}
```

### 问题6：代码是怎么写的？

推荐算法的整体架构和业务逻辑是自己设计的，包括工厂模式的封装、多种算法的选择和混合策略。协同过滤的矩阵构建和相似度计算是自己实现的。机器学习部分调用了 Weka 这个开源库来做模型训练和预测，但数据预处理、特征工程、模型参数调优和模型持久化管理这些都是自己写的。业务层面的多阶段推荐策略（需求画像 → 模型预测 → 协同过滤 → 热度兜底）也是根据实际场景自己设计的。

### 问题7：用户推荐是怎么做的？

用户推荐用于帮助当前用户发现可能感兴趣的其他用户，实现在 `YonghuController` 的 `/recommend` 接口。算法综合考虑三个维度：

1. **技能匹配度**：找到在当前用户关注的分类下发布了"技能"的用户，基础分 3.0
2. **需求相似度**：通过余弦相似度衡量两个用户的需求偏好是否接近，权重 ×2.0
3. **信誉值加成**：用户的信誉值（`xinyuzhishu / 100`）作为额外加分项

最终按综合得分排序，默认推荐 6 位用户。

### 加分话术

- **主动提到局限性**：「目前的实现是实时计算相似度，数据量大时性能可能不够，后续可以考虑离线预计算相似度矩阵。」
- **提到数据来源**：「收藏和点赞行为统一存储在 storeup 表中，收藏的权重是点赞的两倍，体现了不同行为的兴趣强度差异。」
- **参数调优**：「机器学习模型的参数是参考了 Weka 官方文档的推荐配置，然后根据实际数据量做了调整。」
- **多场景应用**：「推荐算法不仅用于技能推荐，还复用了需求画像做学习资料的跨领域推荐，以及综合信誉值的用户推荐。」
