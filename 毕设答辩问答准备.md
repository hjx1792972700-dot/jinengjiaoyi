## 一、项目概述类问题

### Q：简单介绍一下你的毕设做的是什么？

这是一个**技能交易平台**，解决的是人与人之间技能互换的需求。比如一个会弹吉他的人想学摄影，一个会摄影的人想学吉他，他们可以通过这个平台发布自己的技能和需求，系统会智能匹配，双方互相交换技能，实现"以技换技"。

平台主要功能包括：

- **技能市场**：用户发布技能和需求，浏览和搜索其他人的技能
- **智能匹配与推荐**：基于协同过滤和机器学习算法推荐感兴趣的技能
- **交换工作流**：申请 → 选择技能 → 确认交换 → 记录 → 评价，完整闭环
- **学习资料**：付费/免费的学习资源分享
- **技能社区**：论坛式交流、好友私聊
- **AI 智能客服**：接入百度千帆大模型，支持流式问答
- **会员体系**：VIP 充值、积分信誉

系统分为**管理端**（后台管理）和**用户端**（前台使用）两套界面。

### Q：你的项目有什么创新点/亮点？

1. **混合推荐算法**：不是单纯的协同过滤，而是结合了用户协同过滤、物品协同过滤和机器学习模型（随机森林/神经网络）的混合推荐，解决了冷启动和推荐数量不足的问题。
2. **完整的交换工作流**：从申请到确认到评价，有完整的状态机流转（待审核→待确认→已完成/拒绝），而不是简单的 CRUD。
3. **AI 大模型集成**：接入百度千帆大模型 API，支持流式输出（SSE），用户可以实时看到 AI 的回答过程。
4. **信誉体系**：用户有信誉指数，交换后互相评价会影响信誉分，保证了交易的可信度。

### Q：前后端是怎么交互的？

前后端通过 **RESTful API + JSON** 交互：

1. 前端 Vue 通过 `axios` 发送 HTTP 请求（GET/POST）
2. 请求头携带 `Token` 字段用于身份认证
3. 后端 Controller 接收请求，处理业务逻辑，返回统一格式 `R` 对象（包含 code、msg、data）
4. 前端根据返回的 code 判断成功/失败，展示数据或提示错误

认证流程：

- 用户登录 → 后端生成 Token 存入数据库 → 返回 Token 给前端
- 前端存入 `localStorage`，后续每个请求都在 Header 中带上 Token
- 后端 `AuthorizationInterceptor` 拦截器校验 Token 有效性
- 标注了 `@IgnoreAuth` 注解的接口不需要 Token（如登录、注册、文件上传）

### Q：你的项目分了哪些层？

```
com/
├── controller/     ← 控制层：接收请求、参数校验、调用Service、返回结果
├── service/        ← 业务逻辑层（接口）
│   └── impl/       ← 业务逻辑层（实现）
├── dao/            ← 数据访问层（MyBatis-Plus Mapper 接口）
├── entity/         ← 实体类（对应数据库表）
│   ├── view/       ← 视图实体（关联查询结果）
│   ├── vo/         ← 值对象（前端传参）
│   └── model/      ← 模型对象
├── config/         ← 配置类（MyBatis、拦截器注册等）
├── interceptor/    ← 拦截器（Token 认证、CORS）
├── annotation/     ← 自定义注解（@IgnoreAuth、@SysLog）
├── algorithm/      ← 算法模块（推荐算法工厂）
├── llm/            ← AI 大模型集成
└── utils/          ← 工具类
```

对应的 mapper XML 在 `resources/mapper/` 下，和 DAO 接口一一对应。

| **评论与收藏**  | `discussjinengxuqiu`、`discussxuexiziliao`、`storeup`                            | 技能评论、资料评论、收藏                                   |

| **会员**     | `huiyuanchongzhi`                                                              | 会员充值订单                                         |
| **系统**     | `config`、`syslog`、`sensitivewords`                                             | 系统配置、操作日志、敏感词                                  |


### Q：为什么技能需求和技能发布放在同一张表？

`jinengxuqiu` 表通过 `leixing` 字段区分「需求」和「技能」两种类型。这样做的好处是：

1. **结构完全一致**：需求和技能的字段（标题、分类、描述、费用、封面等）基本相同，没必要拆两张表
2. **方便匹配**：推荐和搜索时可以在同一张表里做跨类型查询
3. **减少冗余**：评论、收藏、点赞等关联表只需要关联一张表

### Q：你的数据库版本是怎么管理的？

使用 **Liquibase** 进行数据库版本管理：

- 所有数据库变更写成 XML changelog 文件，放在 `resources/db/changelog/` 目录下
- `db.changelog-master.xml` 是主文件，按顺序 include 所有变更文件
- 应用启动时 Liquibase 自动检测并执行未执行过的变更
- 好处是：数据库变更可追溯、团队协作不冲突、支持回滚

## 四、核心功能实现类问题

### Q：交换工作流是怎么实现的？

交换工作流通过 `jiaohuanshenqing` 表的 `sfsh`（审核状态）字段驱动状态流转：

```
用户A发布技能需求 → 用户B发起交换申请（sfsh=待审核）
  ↓
用户A查看申请，选择B的某个技能（sfsh=待确认）
  ↓
用户A确认交换（sfsh=已完成）→ 自动生成交换记录（jiaohuanjilu表）
  或
用户A拒绝交换（sfsh=否）→ 记录拒绝原因
```

关键控制：

- **防重复申请**：同一用户对同一需求只能有一个未拒绝的申请
- **不能自己申请自己**：后端校验申请人和发布者不能是同一人
- **确认时自动创建交换记录**：把申请信息（双方账号、技能、分类、封面等）复制到交换记录表

### Q：文件上传是怎么做的？

1. 前端通过 Element UI 的 Upload 组件选择文件，POST 到 `/file/upload`
2. 后端接收 `MultipartFile`，生成唯一文件名（时间戳 + 随机数 + 原扩展名）
3. 保存到 `classpath:static/upload/` 目录下
4. 同时复制一份到源码目录（开发时方便调试）
5. 返回文件名给前端，前端保存文件名到对应的业务表字段
6. 文件大小限制 **100MB**（`application.yml` 配置）

安全措施：下载接口做了路径校验，禁止 `..` 路径穿越。

### Q：AI 智能客服是怎么实现的？

接入了**百度千帆大模型 API**（ERNIE 4.5），实现流程：

1. 前端发送用户问题到后端 `/askai/stream` 接口
2. 后端构建请求体（OpenAI 兼容格式），调用千帆 API
3. 使用 **SSE（Server-Sent Events）** 流式传输，后端逐块读取 AI 响应并实时推送给前端
4. 前端收到 SSE 事件后逐字显示，实现打字机效果
5. 聊天记录存入 `chat` 表

技术细节：

- API Key 存在数据库 `config` 表中，通过 `ConfigUtil` 读取
- 支持上传文件（图片转 Base64、PDF 提取文本）作为上下文
- HTTPS 连接使用 TLSv1.2

### Q：登录认证是怎么做的？

采用 **Token 认证机制**（非 Session、非 JWT）：

1. 用户登录时，后端验证账号密码，生成一个 Token 字符串存入 `token` 表（关联 userId、role、tableName、过期时间）
2. 返回 Token 给前端，前端存入 `localStorage`
3. 后续请求在 Header 中携带 `Token` 字段
4. `AuthorizationInterceptor` 拦截器拦截所有请求（除了登录、注册、文件上传等 `@IgnoreAuth` 接口）
5. 根据 Token 查库，有效则放行并把用户信息写入 Session 属性，无效则返回 401

前端登录还加了**滑块验证码**防止暴力破解。

## 五、推荐算法类问题

> 详细说明参见《推荐算法说明文档.md》

### Q：推荐算法用在了哪些地方？


| 接口                            | Controller            | 功能                             |
| ----------------------------- | --------------------- | ------------------------------ |
| `/jinengxuqiu/trainRecommend` | JinengxuqiuController | 训练技能推荐模型（NCF 深度学习）             |
| `/jinengxuqiu/autoSort2`      | JinengxuqiuController | 给用户推荐技能（混合推荐：需求侧协同过滤 + 机器学习模型） |
| `/yonghu/recommend`           | YonghuController      | 推荐技能达人（基于需求向量的余弦相似度 + 信誉加权）    |


### Q：autoSort2 的推荐逻辑是什么？

`autoSort2` 是最核心的推荐接口，分两个阶段：

**阶段1 — 需求侧协同过滤：**

1. 提取所有「需求」类型的技能，按用户分组构建需求向量（维度是技能分类）
2. 计算当前用户和其他用户的余弦相似度，取 Top 20 相似用户
3. 从相似用户的收藏（权重×2）和点赞（权重×1）中提取推荐候选，排除已收藏的
4. 补充同分类下的热门技能（按点击量排序）

**阶段2 — 机器学习模型补充：**

1. 如果阶段1的推荐数量不够，加载已训练的 `jinengxuqiu.model`
2. 从收藏记录构建用户-物品矩阵
3. 用模型预测 + 协同过滤补充，合并到最终结果

### Q：训练数据是怎么来的？

- **用户特征**：从 `yonghu` 表提取（性别、年龄等）
- **物品特征**：从 `jinengxuqiu` 表提取（标题、分类等）
- **交互行为**：从 `storeup`（收藏表）中取 `type=1`（收藏）且 `tablename="jinengxuqiu"` 的记录，每条记录变成一个 `UserBehavior(userId, itemId, 1.0)`

## 六、安全与性能类问题

### Q：你做了哪些安全措施？

1. **Token 认证**：所有敏感接口需要有效 Token，过期自动失效
2. **CORS 配置**：拦截器统一处理跨域，只允许合法的 Origin
3. **文件上传安全**：下载接口禁止路径穿越（`..` 检测）
4. **敏感词过滤**：`sensitivewords` 表维护敏感词库，内容发布时检测
5. **SQL 注入防护**：MyBatis-Plus 使用参数化查询，`SQLFilter` 工具类过滤特殊字符
6. **操作日志**：`@SysLog` 注解 + AOP 记录关键操作（谁在什么时间做了什么）
7. **滑块验证码**：登录时防暴力破解
8. **逻辑删除**：MyBatis-Plus 配置逻辑删除（`-1` 为已删除，`0` 为正常），数据不会真正删除

### Q：你考虑过性能问题吗？

1. **模型缓存**：推荐模型加载后缓存在内存（`ConcurrentHashMap`），不需要每次都从磁盘读取
2. **矩阵缓存**：用户-物品矩阵和物品-用户矩阵都用 `ConcurrentHashMap` 缓存
3. **数据库索引**：Liquibase 建表时对常用查询字段建了索引
4. **分页查询**：列表接口都使用 MyBatis-Plus 分页插件，不会一次加载全部数据
5. **文件上传限制**：限制最大 100MB，防止恶意大文件上传

## 七、开发过程类问题

### Q：开发过程中遇到了什么困难？怎么解决的？

**困难1：推荐算法冷启动**

- 问题：新用户没有行为数据，协同过滤无法工作
- 解决：采用混合推荐策略，模型可以利用用户本身的特征（年龄、性别等）做预测，不完全依赖行为数据；同时补充热门内容作为兜底

**困难2：交换工作流的状态管理**

- 问题：交换申请有多个状态（待审核、待确认、已完成、拒绝），状态流转容易出错
- 解决：在每个状态变更的接口里严格校验当前状态，比如只有"待审核"状态才能选择技能，只有"待确认"状态才能确认/拒绝

**困难3：AI 大模型的流式输出**

- 问题：普通 HTTP 请求是一次性返回，用户要等很久才能看到结果
- 解决：使用 SSE（Server-Sent Events）协议，后端逐块读取大模型响应并实时推送到前端，用户可以看到逐字输出的效果

**困难4：数据库频繁变更**

- 问题：开发过程中经常要加字段、改数据、加表
- 解决：引入 Liquibase 做数据库版本管理，所有变更写成 changelog 文件，启动时自动执行，避免手动执行 SQL 出错

### Q：你的项目还有哪些不足？后续怎么改进？

1. **推荐算法可以优化**：目前是实时计算相似度，数据量大时性能会下降，后续可以改为离线预计算相似度矩阵，定时更新
2. **没有做消息推送**：交换申请的状态变更目前需要用户主动刷新才能看到，后续可以接入 WebSocket 做实时通知
3. **安全性可以加强**：Token 目前是明文存储，可以改为 JWT 并加入刷新机制；密码可以加 bcrypt 加盐哈希
4. **前端可以升级**：目前用的 Vue 2 + Element UI，后续可以升级到 Vue 3 + Element Plus，提升性能和开发体验
5. **缺少单元测试**：目前没有写测试用例，后续可以补充 JUnit 单元测试和接口测试
