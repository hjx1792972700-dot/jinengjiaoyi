# 毕设答辩问答准备

> 项目名称：基于协同过滤的技能交易系统的设计与实现

## 一、项目概述类问题

### Q：简单介绍一下你的毕设做的是什么？

这是一个**校园技能交易平台**，解决的是大学生之间技能互换的需求。比如一个会弹吉他的人想学摄影，一个会摄影的人想学吉他，他们可以通过这个平台发布自己的技能和需求，系统会智能匹配，双方互相交换技能，实现"以技换技"。

平台主要功能包括：

- **技能市场**：用户发布技能供给和学习需求，浏览和搜索其他人的技能
- **智能推荐**：基于协同过滤和神经协同过滤（NCF）混合推荐算法，推荐匹配的技能和用户
- **交换工作流**：申请 → 选择技能 → 确认交换 → 生成记录，完整的状态机闭环
- **学习资料**：付费/免费的学习资源分享与购买，VIP 用户可免费获取
- **技能社区**：论坛式交流、WebSocket 实时私聊、好友系统
- **AI 智能客服**：接入大模型 API，支持 SSE 流式问答
- **VIP 会员体系**：充值、套餐选择、会员特权（发布技能、免费资料）
- **管理后台**：数据统计可视化（ECharts）、内容审核、用户管理

系统分为**管理端**（后台管理，端口 8082）和**用户端**（前台使用，端口 8081）两套 Vue 前端界面，共用一个 Spring Boot 后端（端口 48081）。

### Q：你的项目有什么创新点/亮点？

1. **混合推荐算法**：不是单纯的协同过滤，而是结合了需求侧协同过滤（基于用户需求向量的余弦相似度）和 NCF 神经协同过滤模型（Weka RandomForest/MLP 训练），解决了冷启动和推荐数量不足的问题，不足时还有热门内容兜底。
2. **完整的交换工作流**：从申请到确认到记录生成，有完整的状态机流转（待审核→待确认→已完成/拒绝），包含防重复申请、禁止自我申请、双向技能选择等业务约束。
3. **AI 大模型集成**：支持 SSE（Server-Sent Events）流式输出，用户可以实时看到 AI 的回答过程，聊天历史持久化存储。
4. **VIP 差异化服务**：VIP 用户独享技能发布权限和学习资料免费访问，形成完整的付费闭环。
5. **WebSocket 实时通信**：基于 JSR-356 的点对点私信系统，支持心跳检测和在线状态管理。
6. **Liquibase 数据库版本管理**：所有表结构和测试数据通过 15 个 changelog 文件（160 个 changeSet）管理，确保在任何新环境中可一键重建完整数据库。

### Q：前后端是怎么交互的？

前后端通过 **RESTful API + JSON** 交互：

1. 前端 Vue 通过 `axios` 发送 HTTP 请求（GET/POST）
2. 请求头携带 `Token` 字段用于身份认证
3. 后端 Controller 接收请求，处理业务逻辑，返回统一格式 `R` 对象（包含 code、msg、data）
4. 前端根据返回的 code 判断成功/失败，展示数据或提示错误

**认证流程**：

- 用户登录 → 后端生成 32 位随机 Token 存入 `token` 表（有效期 1 小时）→ 返回给前端
- 前端存入 `localStorage`，后续每个请求都在 Header 中带上 `Token`
- 后端 `AuthorizationInterceptor` 拦截器校验 Token 有效性和过期时间
- 标注了 `@IgnoreAuth` 的接口免认证（如登录、注册、公开浏览）

**实时通信**：

- WebSocket 通过 `@ServerEndpoint("/ws")` 建立连接，query 参数传递 `user_id` 和 `to_id`
- 使用 `ConcurrentHashMap` 管理在线连接，支持点对点消息转发和 ping/pong 心跳

### Q：你的项目分了哪些层？

```
com/
├── controller/        ← 控制层：接收请求、参数校验、调用 Service
├── service/impl/      ← 业务逻辑层（交换工作流、VIP 逻辑、推荐等）
├── dao/               ← 数据访问层（MyBatis-Plus Mapper 接口）
├── entity/            ← 实体类（对应数据库表）
│   ├── view/          ← 视图实体（关联查询结果）
│   └── vo/            ← 值对象（前端传参）
├── config/            ← 配置类（MyBatis 分页、拦截器注册、CORS）
├── interceptor/       ← 拦截器（Token 认证）
├── annotation/        ← 自定义注解（@IgnoreAuth、@SysLog）
├── algorithm/recommend/ ← 推荐算法工厂（CF + NCF + 模型管理）
├── ws/                ← WebSocket 消息管理器
└── utils/             ← 工具类（协同过滤、加密、脱敏、SQL过滤等）
```

对应的 MyBatis XML 映射文件在 `resources/mapper/` 下，与 DAO 接口一一对应，使用动态 SQL 实现灵活查询和统计。

## 三、数据库设计类问题

### Q：你的数据库怎么设计的？有多少张表？

数据库名 `jinengjiaoyi`，核心业务表按功能分类如下：

| 分类 | 表名 | 说明 |
|------|------|------|
| **用户与认证** | `yonghu`、`users`、`token` | 普通用户（含VIP、信誉指数）、管理员、登录令牌 |
| **技能交易核心** | `jinengfenlei`、`jinengxuqiu`、`jiaohuanshenqing`、`jiaohuanjilu` | 8 大技能分类、技能/需求统一表（`leixing` 字段区分）、交换申请（状态机驱动）、交换记录 |
| **学习资料** | `ziliaoleixing`、`xuexiziliao`、`ziliaogoumai` | 8 大资料类型、学习资料（含价格）、购买订单（VIP 免费） |
| **社区与社交** | `forumtype`、`forum`、`friend`、`chatmessage`、`messages` | 7 个论坛分类、帖子（支持回帖）、好友关系、WebSocket 私聊、留言反馈 |
| **评论与收藏** | `discussjinengxuqiu`、`discussxuexiziliao`、`storeup` | 技能评论、资料评论、多态收藏（支撑推荐算法输入） |
| **AI 与客服** | `chat`、`chathelper` | AI 问答历史、预设常见问答 |
| **资讯** | `newstype`、`news` | 资讯分类、公告资讯 |
| **会员** | `huiyuanchongzhi` | VIP 充值订单（月度/季度/年度/连续包月） |
| **系统** | `config` | 系统配置参数 |

### Q：为什么技能需求和技能发布放在同一张表？

`jinengxuqiu` 表通过 `leixing` 字段区分「需求」和「技能」两种类型。这样做的好处是：

1. **结构完全一致**：需求和技能的字段（标题、分类、描述、费用、封面等）基本相同，没必要拆两张表
2. **方便匹配**：推荐和搜索时可以在同一张表里做跨类型查询
3. **减少冗余**：评论、收藏、点赞等关联表只需要关联一张表

### Q：你的数据库版本是怎么管理的？

使用 **Liquibase** 进行数据库版本管理：

- 所有数据库变更写成 XML changelog 文件，放在 `resources/db/changelog/` 目录下
- `db.changelog-master.xml` 是主文件，按顺序 include 了 **15 个 changelog 文件**（共 **160 个 changeSet**）
- 应用启动时 Liquibase 自动检测并执行未执行过的变更
- 使用 `INSERT IGNORE`、`CREATE TABLE IF NOT EXISTS`、`preConditions` 等机制保证幂等性
- 好处是：数据库变更可追溯、团队协作不冲突、新环境一键重建

### Q：storeup 收藏表为什么设计成多态的？

`storeup` 表通过 `tablename`（关联表名）和 `refid`（关联记录 ID）实现**多态收藏**，可以收藏技能、学习资料、帖子等多种内容。这样设计的关键作用是：

1. **统一收藏入口**：前端只需要调用一个 API，传不同的 `tablename` 参数
2. **推荐算法数据源**：协同过滤算法直接从这张表构建用户-物品行为矩阵，`type` 字段区分收藏/点赞/评论等不同行为权重

## 四、核心功能实现类问题

### Q：交换工作流是怎么实现的？

交换工作流通过 `jiaohuanshenqing` 表的 `sfsh`（审核状态）字段驱动状态流转：

```
用户B浏览用户A发布的技能需求 → 发起交换申请（sfsh=待审核）
  ↓
用户A查看收到的申请（receivedApply），选择B提供的某个技能（sfsh=待确认）
  ↓                                              ↓
用户B确认交换（sfsh=已完成）                     用户B/A拒绝（sfsh=否，附原因）
  ↓
自动生成交换记录（jiaohuanjilu 表，快照双方信息和提供技能）
```

关键控制逻辑（`JiaohuanshenqingController`）：

- **防重复申请** (`checkDuplicate`)：同一用户对同一需求只能有一个 `sfsh != '否'` 的申请
- **不能自己申请自己**：后端校验申请人和发布者不能是同一人
- **selectSkill**：只有 `sfsh='待审核'` 才能选择技能，选择后变为 `待确认`，记录 `xuanzejinengid` 和 `xuanzejinengbiaoti`
- **confirmExchange**：只有 `sfsh='待确认'` 才能确认，确认后自动 INSERT 到 `jiaohuanjilu` 表
- **rejectExchange**：只有 `sfsh='待确认'` 才能拒绝，附拒绝理由写入 `shhf` 字段

### Q：VIP 会员系统是怎么实现的？

VIP 系统涉及 `yonghu` 表的 `vip`/`vip_expire` 字段和 `huiyuanchongzhi` 充值表：

**充值流程**（`HuiyuanchongzhiController`）：
1. `createOrder`：用户选择套餐（月度 30 / 季度 80 / 年度 200 / 连续包月 15），创建充值订单
2. `payCallback`：支付回调，更新订单状态为"已支付"，同时更新 `yonghu` 表的 VIP 状态和到期时间
3. `vipStatus`：查询 VIP 状态，自动处理过期降级

**VIP 特权体现在代码中**：
- `JinengxuqiuController.add`：发布"技能"类型时校验 `vip=='是'`，非 VIP 只能发布"需求"
- `ZiliaogoumaiController.checkPurchased`：VIP 用户直接返回"已购买"，无需付费

### Q：文件上传是怎么做的？

1. 前端通过 Element UI 的 Upload 组件选择文件，POST 到 `/file/upload`（`@IgnoreAuth`）
2. 后端接收 `MultipartFile`，生成唯一文件名（`时间戳_随机数.扩展名`）
3. 保存到 `classpath:static/upload/` 目录下
4. 返回文件路径给前端，前端保存到对应业务表字段
5. 文件大小限制 **100MB**（`application.yml` 中 `max-file-size` 配置）
6. 下载接口做了路径穿越防护（禁止 `..`）
7. 额外提供 `/file/encrypt` 和 `/file/decrypt` 接口（DES/AES 加密）

### Q：AI 智能客服是怎么实现的？

通过 `CommonController` 中的 `/{model}/askai` 和 `/{model}/askai/stream` 接口实现：

1. 前端发送用户问题到 `/askai/stream` 接口
2. 后端从 `ChatService` 加载当前用户的历史对话作为上下文
3. 构建请求体（OpenAI 兼容格式），调用大模型 API
4. 使用 **SSE（Server-Sent Events）** 流式传输，后端逐块读取 AI 响应并实时推送给前端
5. 前端收到 SSE 事件后逐字显示，实现打字机效果
6. 聊天记录持久化到 `chat` 表
7. 预设的常见问答存储在 `chathelper` 表中，用户可快速选择

### Q：登录认证是怎么做的？

采用 **自定义 Token 认证机制**：

1. 用户登录时验证账号密码，`TokenServiceImpl.generateToken` 生成 32 位随机 Token
2. Token 存入 `token` 表（关联 userId、role、tableName、username），**有效期 1 小时**
3. 返回 Token 给前端，前端存入 `localStorage`
4. 后续请求在 HTTP Header 中携带 `Token` 字段
5. `AuthorizationInterceptor` 拦截 `/**`（排除 `/static/**`、`/admin/**`、`/front/**`）
6. 根据 Token 查库，有效则设置 Session 属性（`userId`、`role`、`tableName`、`username`），无效返回 401
7. `@IgnoreAuth` 注解的接口免认证

**双角色体系**：管理员（`users` 表）和普通用户（`yonghu` 表）分别通过 `/users/login` 和 `/yonghu/login` 登录，使用不同的 Token 存储键（`Token` vs `frontToken`）。

### Q：WebSocket 私信是怎么实现的？

通过 `WebSocketManager` 类实现（`com.ws.WebSocketManager`）：

1. 使用 `@ServerEndpoint("/ws")` 注解定义 WebSocket 端点
2. 连接时通过 query 参数传递 `user_id` 和 `to_id`
3. 用 `ConcurrentHashMap` 维护所有在线连接的映射
4. 收到消息时，查找目标用户的连接并转发；目标不在线则广播或存储
5. 支持 `ping`/`pong` 心跳检测保持连接
6. 消息同时存入 `chatmessage` 表，支持已读/未读状态标记（`markRead`）

## 五、推荐算法类问题

### Q：推荐算法用在了哪些地方？

| 接口 | Controller | 功能 |
|------|-----------|------|
| `/jinengxuqiu/trainRecommend` | JinengxuqiuController | 训练 NCF 推荐模型（Weka RandomForest/MLP） |
| `/jinengxuqiu/autoSort2` | JinengxuqiuController | 给用户推荐技能（混合推荐：需求侧协同过滤 + NCF 模型） |
| `/yonghu/recommend` | YonghuController | 推荐技能达人（需求向量余弦相似度 + 信誉指数加权） |
| `/xuexiziliao/autoSort` | XuexiziliaoController | 推荐学习资料（基于收藏行为的协同过滤） |

### Q：autoSort2 的推荐逻辑是什么？

`autoSort2` 是最核心的推荐接口，分两个阶段（`JinengxuqiuController`）：

**阶段 1 — 需求侧协同过滤：**

1. 提取所有 `leixing='需求'` 的记录，按用户分组，以 `jinengfenlei`（技能分类）为维度构建需求向量
2. 计算当前用户和其他用户的**余弦相似度**，取 Top N 相似用户
3. 从相似用户的收藏记录（`storeup` 表，`type=1` 收藏权重 ×2，`type=21` 点赞权重 ×1）中提取推荐候选
4. 排除当前用户已收藏/已浏览的内容，补充同分类下热门技能

**阶段 2 — NCF 模型补充：**

1. 如果阶段 1 推荐数量不足，加载已训练的 `jinengxuqiu.model`
2. `RecommendAlgorithmFactory.loadModel()` 从 `algorithm/recommend/models/` 目录读取序列化模型
3. 从 `storeup` 构建用户-物品矩阵，用模型预测 + 协同过滤补充
4. 合并去重后通过 `ORDER BY FIELD(id, 推荐ID列表)` 排序返回
5. 如果还不够，用热门列表（按 `clicknum` 排序）兜底

### Q：trainRecommend 训练过程是怎样的？

1. 从 `yonghu` 表提取用户特征（ID、性别、年龄等）
2. 从 `jinengxuqiu` 表提取物品特征（ID、标题、分类等）
3. 从 `storeup` 表提取用户-物品交互行为（`tablename='jinengxuqiu'`），每条记录变成 `UserBehavior(userId, itemId, score)`
4. 通过 `RecommendAlgorithmFactory` 构建训练实例，使用 Weka 的 RandomForest 或 MultilayerPerceptron 训练
5. 训练好的模型序列化保存到 `algorithm/recommend/models/jinengxuqiu.model`

### Q：余弦相似度的公式是什么？

$$\text{similarity}(A, B) = \frac{A \cdot B}{\|A\| \times \|B\|} = \frac{\sum_{i=1}^{n} A_i \times B_i}{\sqrt{\sum_{i=1}^{n} A_i^2} \times \sqrt{\sum_{i=1}^{n} B_i^2}}$$

在本系统中，向量 A 和 B 是两个用户的**需求分类向量**，每个维度代表一个技能分类（学习、生活、编程、设计等 8 个分类），值为该用户在该分类下发布的需求数量。相似度越接近 1，说明两个用户的需求偏好越相似。

## 六、安全与性能类问题

### Q：你做了哪些安全措施？

1. **Token 认证**：所有敏感接口需要有效 Token，1 小时自动过期
2. **CORS 跨域控制**：`InterceptorConfig` 统一配置跨域策略，`AuthorizationInterceptor` 对 OPTIONS 预检请求直接放行
3. **文件上传安全**：时间戳+随机数重命名防猜测，下载接口禁止 `..` 路径穿越，提供 DES/AES 加密
4. **SQL 注入防护**：MyBatis-Plus 使用 `EntityWrapper` 参数化查询，`SQLFilter` 工具类过滤排序字段中的危险关键字
5. **滑块验证码**：前端登录时防暴力破解
6. **逻辑删除**：MyBatis-Plus 配置逻辑删除（`-1` 已删除，`0` 正常），数据不会真正删除
7. **业务校验**：交换申请防重复、禁止自我申请、VIP 权限校验、审核状态严格流转

### Q：你考虑过性能问题吗？

1. **推荐模型缓存**：`RecommendAlgorithmFactory` 使用 `ConcurrentHashMap` 缓存已加载的模型和用户-物品矩阵，避免重复磁盘 IO
2. **分页查询**：所有列表接口使用 MyBatis-Plus `PaginationInterceptor` 分页插件
3. **WebSocket 连接管理**：使用 `ConcurrentHashMap` 管理在线连接，O(1) 查找目标用户
4. **数据库索引**：Liquibase 建表时对常用查询字段（如 `typename`、`yonghuzhanghao`）建了索引
5. **文件上传限制**：限制最大 100MB，防止恶意大文件
6. **SSE 流式输出**：AI 问答使用流式传输，避免长时间等待阻塞

## 七、开发过程类问题

### Q：开发过程中遇到了什么困难？怎么解决的？

**困难 1：推荐算法冷启动**

- 问题：新用户没有收藏/点赞等行为数据，协同过滤无法工作
- 解决：采用**三级回退策略** —— 优先用协同过滤推荐，不足时用 NCF 模型补充（模型可利用用户特征做预测），最后用热门内容兜底（按点击量排序）

**困难 2：交换工作流的状态管理**

- 问题：交换申请有多个状态（待审核、待确认、已完成、拒绝），并发操作可能导致状态不一致
- 解决：在每个状态变更接口里**严格校验当前状态**——只有"待审核"才能 selectSkill，只有"待确认"才能 confirmExchange/rejectExchange，防止非法状态跳转

**困难 3：AI 大模型的流式输出**

- 问题：普通 HTTP 请求是一次性返回，长文本生成时用户要等很久才能看到结果
- 解决：使用 **SSE（Server-Sent Events）** 协议，后端通过 `SseEmitter` 逐块读取大模型响应并实时推送到前端，前端监听事件逐字渲染

**困难 4：数据库频繁变更**

- 问题：开发过程中经常要加字段、改数据、加表，手动执行 SQL 容易遗漏或出错
- 解决：引入 **Liquibase** 做数据库版本管理，所有变更写成 XML changelog，使用 `preConditions` 保证幂等性，启动时自动执行未跑过的变更

**困难 5：多端前端与后端的协同部署**

- 问题：用户端和管理端是两套独立的 Vue 项目，开发时端口不同，部署时需要嵌入后端
- 解决：开发时通过 `vue.config.js` 配置代理（proxy）转发 API 请求到后端；生产环境通过 `npm run build` 构建后复制 `dist` 到 Spring Boot 的 `resources/admin/dist` 和 `resources/front/dist`，后端通过 `addResourceHandlers` 统一托管静态资源

### Q：你的项目还有哪些不足？后续怎么改进？

1. **安全性加强**：Token 目前存在数据库中，可以改为 JWT 减少数据库查询；密码目前明文存储，应加 bcrypt 加盐哈希
2. **推荐算法优化**：目前是实时计算相似度，数据量大时性能会下降，后续可改为离线预计算相似度矩阵，定时更新
3. **消息推送完善**：交换申请状态变更时，可以通过已有的 WebSocket 通道主动推送通知，而不是等用户刷新
4. **前端升级**：目前用的 Vue 2 + Element UI，后续可以升级到 Vue 3 + Element Plus，提升性能和开发体验
5. **缺少自动化测试**：目前没有写测试用例，后续可以补充 JUnit 单元测试和接口集成测试
6. **支付系统对接**：当前使用模拟支付，后续可接入微信支付/支付宝真实支付接口
