# 基于协同过滤的技能交易系统 — 用户使用文档

## 一、系统概述

本系统是一个基于 Spring Boot + Vue 前后端分离架构的**技能交易平台**，核心目标是让用户能够发布自己拥有的技能和需要的技能，通过平台的智能推荐与匹配完成**技能交换**，同时提供**学习资料购买**功能辅助用户提升技能水平。

系统采用**协同过滤算法**作为推荐引擎的核心，结合余弦相似度、用户行为矩阵和基于 Weka 的多层感知器（MLP）模型，为每位用户提供个性化的技能与资料推荐。

### 1.1 技术栈

| 层级 | 技术 |
|------|------|
| 后端框架 | Spring Boot 2.2 + MyBatis-Plus |
| 前端框架 | Vue 2 + Element UI（双端 SPA：用户端 + 管理端） |
| 数据库 | MySQL 8.x + Liquibase 数据库版本管理 |
| 推荐算法 | 余弦相似度 + 用户协同过滤 + 物品协同过滤 + Weka MLP |
| AI 对话 | 百度千帆大模型（ernie-4.5-0.3b），支持 SSE 流式输出 |
| 实时通信 | WebSocket 私聊 |
| 数据可视化 | ECharts 统计图表 |

### 1.2 用户角色

| 角色 | 说明 |
|------|------|
| **普通用户** | 注册后可发布需求、浏览技能、发起交换、购买资料 |
| **VIP 会员** | 在普通用户基础上可发布"技能"类型列表、免费下载付费资料 |
| **管理员** | 后台管理，审核内容、管理用户、查看统计数据 |

## 二、核心功能一：技能交换

技能交换是本系统的核心业务，完整流程涵盖"发布 → 审核 → 推荐 → 申请 → 选择 → 确认"六个阶段。

### 2.1 业务流程

```
用户发布技能/需求 ──→ 管理员审核 ──→ 审核通过，公开展示
                                         │
                         ┌────────────────┘
                         ▼
              其他用户浏览（推荐算法排序）
                         │
                         ▼
              发起交换申请（jiaohuanshenqing）
                         │
                         ▼
              发布者选择对方技能（selectSkill）
                   状态：待审核 → 待确认
                         │
                         ▼
              申请者确认交换（confirmExchange）
                   状态：待确认 → 已完成
                         │
                         ▼
              系统生成交换记录（jiaohuanjilu）
```

#### 详细步骤说明

**第一步：发布技能/需求**

用户在前端提交技能或需求信息，系统自动标记为"待审核"状态。其中**技能类型的发布仅限 VIP 用户**，普通用户只能发布"需求"类型。每条发布包含标题、技能分类、详细描述、联系方式等字段。

**第二步：管理员审核**

管理员在后台通过"技能审核"或"需求审核"页面分别审核技能和需求内容，设置审核状态为"是"（通过）或"否"（驳回），并可附上审核回复。只有审核通过的内容才会在前端列表和详情页中展示。管理端首页的"待办事项"区域会实时显示待审核技能和待审核需求的数量，点击可直接跳转至对应的审核页面。

**第三步：智能推荐展示（核心算法，详见 2.2 节）**

系统根据当前用户的需求偏好，通过多阶段混合推荐算法对审核通过的技能进行个性化排序推荐。

**第四步：发起交换申请**

用户在技能详情页发起交换申请。系统会进行以下校验：

- 不允许对自己的技能发起申请
- 同一用户对同一技能不允许重复申请（已有非"驳回"状态的申请时拒绝）

**第五步：发布者选择技能**

收到申请后，技能发布者在"收到的申请"页面查看申请列表，选择申请者拥有的某项技能作为交换标的。此操作将申请状态从"待审核"变为"待确认"，并记录所选技能的 ID 和标题。

**第六步：申请者确认交换**

申请者在"我的申请"中看到"待你确认"的申请，确认后状态变为"已完成"，同时系统自动创建一条**交换记录**（`jiaohuanjilu`），记录双方提供的技能信息作为交易凭证。任何一方也可以选择"拒绝"，此时状态变为"否"。

### 2.2 技能推荐算法详解

技能推荐是系统的技术核心，实现在 `JinengxuqiuController.autoSort2` 方法中，采用**多阶段混合推荐策略**。

#### 第一阶段：基于需求画像的用户相似度推荐

**算法思路**：根据用户发布的"需求"内容，构建每个用户的兴趣向量，通过余弦相似度找到相似用户，再将相似用户收藏/点赞的技能推荐给当前用户。

**Step 1 — 构建用户需求向量**

从所有类型为"需求"的记录中，统计每个用户在各"技能分类"上的发布数量，形成一个稀疏向量：

```
用户A的需求向量: {Java开发: 3, Python编程: 1, UI设计: 2}
用户B的需求向量: {Java开发: 2, 数据分析: 1}
用户C的需求向量: {Python编程: 2, UI设计: 1, 数据分析: 3}
```

**Step 2 — 余弦相似度计算**

对当前用户与其他所有用户计算余弦相似度，公式为：

$$\text{similarity}(A, B) = \frac{\sum_{k \in K} A_k \cdot B_k}{\sqrt{\sum_{i} A_i^2} \cdot \sqrt{\sum_{j} B_j^2}}$$

其中 $K$ 为两个用户共同涉及的技能分类集合，分母为各自向量的完整 L2 范数。相似度范围为 $[0, 1]$，值越高表示两个用户的需求偏好越接近。

代码实现：

```java
private double needCosineSimilarity(Map<String, Double> vec1, Map<String, Double> vec2) {
    Set<String> commonKeys = new HashSet<>(vec1.keySet());
    commonKeys.retainAll(vec2.keySet());
    if (commonKeys.isEmpty()) return 0.0;
    double dotProduct = 0.0;
    for (String key : commonKeys) {
        dotProduct += vec1.get(key) * vec2.get(key);
    }
    double norm1 = Math.sqrt(vec1.values().stream().mapToDouble(v -> v * v).sum());
    double norm2 = Math.sqrt(vec2.values().stream().mapToDouble(v -> v * v).sum());
    if (norm1 == 0 || norm2 == 0) return 0.0;
    return dotProduct / (norm1 * norm2);
}
```

**Step 3 — Top-K 相似用户筛选**

取相似度 > 0 的用户，按相似度降序排列，选取前 K 个（K = min(20, 总相似用户数)）。

**Step 4 — 行为加权评分**

遍历相似用户的收藏和点赞行为，对候选技能计算加权得分：

| 行为类型 | 权重 |
|----------|------|
| 收藏（storeup type=1） | similarity × 2 |
| 点赞（storeup type=21） | similarity × 1 |

排除当前用户自己的发布和已交互过的技能。

**Step 5 — 分类热度补充**

对于当前用户需求向量中涉及的分类，额外检索该分类下审核通过的技能，每条给予 +0.5 的补充得分，按点击量排序，最多补充 `limit × 3` 条。

#### 第二阶段：协同过滤 + MLP 模型补充

如果第一阶段推荐结果不足，进入第二阶段：

**Step 6 — 构建用户-物品交互矩阵**

从收藏（type=1）和点赞（type=21）行为中提取用户-技能交互数据，构建稀疏矩阵：

```
         技能1  技能2  技能3  技能4
用户A  [  1.0    0     1.0    0  ]
用户B  [  0     1.0    0     1.0 ]
用户C  [  1.0   1.0    0      0  ]
```

**Step 7 — Weka MLP 模型预测 + User-Based 协同过滤**

系统加载 `jinengxuqiu.model`（通过 `trainRecommend` 接口预先训练的 NCF 模型），对候选技能进行评分预测。当模型预测结果不足时，退化为 User-Based 协同过滤补充：

$$\hat{r}_{u,i} = \sum_{v \in \text{similar}(u)} \text{sim}(u,v) \cdot r_{v,i}$$

其中 $r_{v,i}$ 为相似用户 $v$ 对物品 $i$ 的评分，$\text{sim}(u,v)$ 为用户相似度。

#### 推荐结果合并与兜底

最终推荐列表的构建遵循以下优先级：

1. **需求画像推荐**（阶段一）— 最高优先级
2. **MLP 模型预测**（阶段二）— 次优先级
3. **User-Based 协同过滤**（阶段二）— 补充
4. **热门技能兜底** — 按点击量排序的审核通过技能填充剩余位置

#### 算法流程图

```
当前用户
   │
   ├─→ 提取需求向量（技能分类 → 频次）
   │       │
   │       ├─→ 与其他用户计算余弦相似度
   │       │       │
   │       │       └─→ Top-K 相似用户（K ≤ 20）
   │       │               │
   │       │               └─→ 加权收藏(×2)/点赞(×1)行为 → 候选技能得分
   │       │
   │       └─→ 需求分类内热度补充（+0.5/条）
   │
   ├─→ 结果不足？
   │       │
   │       YES ──→ 构建用户-物品矩阵
   │                   │
   │                   ├─→ MLP 模型预测（如已训练）
   │                   │
   │                   └─→ User-Based CF 补充
   │
   └─→ 仍然不足？
           │
           YES ──→ 热门技能兜底填充（按点击量降序）
```

### 2.3 未登录用户的推荐

未登录或无行为数据的用户，使用 `autoSort`（热度排序），直接按点击量（`clicknum`）降序展示审核通过的技能列表，不涉及个性化计算。

### 3.3 学习资料推荐算法详解

学习资料的推荐实现在 `XuexiziliaoController.autoSort2` 中，算法结构与技能推荐类似但有差异。

#### 算法流程

**Step 1 — 复用需求画像**

与技能推荐相同，从用户发布的"需求"中构建技能分类兴趣向量。这体现了**跨领域推荐**的思想：用户的技能需求偏好同时用于指导资料推荐。

**Step 2 — 余弦相似度计算与 Top-K 用户**

计算方式与技能推荐完全一致（见 2.2 节公式），取相似度 > 0 的前 20 个用户。

**Step 3 — 行为加权评分**

仅使用**收藏行为**（storeup type=1），权重为 `similarity × 2`。与技能推荐不同，资料推荐**不包含点赞行为加权**。

| 行为类型 | 权重 |
|----------|------|
| 收藏（storeup type=1） | similarity × 2 |

**Step 4 — 分类热度补充**

对当前用户需求分类内的资料，每条给予 +0.5 补充得分，按点击量排序。

**Step 5 — 协同过滤补充**

如结果不足，从资料相关的收藏行为构建用户-物品矩阵，执行 User-Based 协同过滤。

**与技能推荐的关键差异**

| 对比维度 | 技能推荐 | 资料推荐 |
|----------|----------|----------|
| 行为类型 | 收藏(×2) + 点赞(×1) | 仅收藏(×2) |
| MLP 模型 | 加载 `jinengxuqiu.model` 预测 | 不使用 MLP 模型 |
| 补充策略 | MLP → User-Based CF → 热门 | User-Based CF → 热门 |
| 兴趣来源 | 需求发布的技能分类 | 同左（跨领域复用） |

## 四、推荐算法体系总览

### 4.1 算法架构

系统的推荐引擎 `RecommendAlgorithmFactory` 是一个静态工厂类，提供三种推荐器：

```
RecommendAlgorithmFactory
   │
   ├── UserItemMatrixRecommender  (User-Based CF)
   │     └── 基于用户-物品交互矩阵的用户协同过滤
   │
   ├── ItemUserMatrixRecommender  (Item-Based CF)
   │     └── 基于物品-用户交互矩阵的物品协同过滤
   │
   └── ModelLoader  (Weka MLP + CF 混合)
         └── 先用 MLP 模型预测，不足时用 CF 补充
```

### 4.2 User-Based 协同过滤

**核心思想**：找到与目标用户兴趣相似的用户群体，将这些用户喜欢但目标用户尚未接触的物品推荐出来。

**相似度计算**：采用余弦相似度，在用户-物品评分向量上计算：

$$\text{sim}(u, v) = \frac{\vec{u} \cdot \vec{v}}{|\vec{u}| \cdot |\vec{v}|} = \frac{\sum_{i \in I_{uv}} r_{u,i} \cdot r_{v,i}}{\sqrt{\sum_{i \in I_u} r_{u,i}^2} \cdot \sqrt{\sum_{i \in I_v} r_{v,i}^2}}$$

其中 $I_{uv}$ 为两用户共同评价过的物品集合，$I_u$、$I_v$ 为各自评价过的全部物品集合。

**预测评分**：对候选物品累加所有相似用户的"评分 × 相似度"：

$$\hat{r}_{u,i} = \sum_{v \in N(u), r_{v,i} > 0} \text{sim}(u,v) \cdot r_{v,i}$$

### 4.3 Item-Based 协同过滤

**核心思想**：计算物品之间的相似度，基于用户历史交互的物品，推荐与之相似的物品。

**物品相似度**：通过物品-用户评分向量的余弦相似度衡量两个物品的相关性。

**预测评分**：对用户交互过的每个物品，将"用户对该物品的评分 × 该物品与候选物品的相似度"累加。

### 4.4 MLP 模型（NCF 变体）

本系统的 NCF 模型基于 Weka 的 `MultilayerPerceptron` 实现，本质是一个以用户特征和物品特征作为输入、预测用户-物品交互得分的前馈神经网络。

**模型结构**

```
输入层（用户特征 + 物品特征拼接）
         │
    全连接层（50 节点，Sigmoid）
         │
    全连接层（25 节点，Sigmoid）
         │
    输出层（预测交互得分）
```

**训练数据构成**

| 数据源 | 特征字段 | 说明 |
|--------|----------|------|
| 用户表 | 性别、年龄 | 用户画像特征 |
| 技能表 | 标题、分类 | 物品内容特征 |
| 收藏表 | userId, itemId | 隐式正反馈（score=1.0） |

**训练超参数**

| 参数 | 值 |
|------|-----|
| 学习率 | 0.3 |
| 动量 | 0.2 |
| 训练轮次 | 500 |
| 隐藏层 | 50,25 |

**推荐策略**：优先使用 MLP 模型预测得分排序，不足时降级为 User-Based CF 补充，保证推荐列表完整。

### 4.5 用户推荐

除技能和资料推荐外，系统还实现了**用户推荐**功能（`YonghuController` 的 `/recommend` 接口），用于向当前用户推荐可能感兴趣的其他用户。

**综合评分模型**：

| 评分维度 | 计算方式 | 说明 |
|----------|----------|------|
| 技能匹配 | 基础分 3.0 | 在用户关注分类下发布了"技能"的用户 |
| 需求相似度 | sim × 2.0 | 需求向量余弦相似度 > 0.3 的用户 |
| 信誉加成 | 信誉值 / 100 | 信誉值越高加分越多 |

默认推荐 6 位用户，不足时按信誉值降序兜底。

### 4.6 多层推荐策略对比

| 策略层级 | 适用场景 | 算法 | 数据依赖 |
|----------|----------|------|----------|
| 需求画像推荐 | 登录用户有需求发布记录 | 需求向量余弦相似度 + 行为加权 | 需求记录 + 收藏/点赞 |
| MLP 模型推荐 | 已训练模型且用户有行为记录 | Weka MultilayerPerceptron | 用户特征 + 物品特征 + 交互数据 |
| User-Based CF | 用户有收藏行为 | 用户-物品矩阵协同过滤 | 收藏行为数据 |
| 热度排序 | 兜底/未登录 | 按点击量降序 | 无 |

## 五、其他功能模块

### 5.1 技能社区（论坛）

用户可在技能社区发帖交流，支持分类浏览、搜索。帖子按板块类型（`forumtype`）组织，支持评论互动。管理员可在后台进行内容管理。

### 5.2 公告资讯

管理员发布系统公告和行业资讯，前端以轮播和分类列表形式展示，支持富文本内容和图片。用户可按资讯类型（`newstype`）筛选浏览。管理端首页也会展示最新公告资讯。

### 5.3 评论与互动系统

技能列表和学习资料均支持用户评论，评论功能包括：

- **星级评分**：用户对技能/资料打分，系统计算平均分更新到列表
- **文字评论**：支持发布和管理员回复
- **点赞/踩**：使用 `tuserids`/`cuserids` 字段记录投票用户，防止重复
- **置顶**：管理员可设置优质评论置顶

### 5.4 收藏与点赞

用户的收藏（type=1）和点赞（type=21）行为统一存储在 `storeup` 表中，通过 `tablename` 字段区分收藏对象（`jinengxuqiu`、`xuexiziliao`、`forum` 等）。这些行为数据同时作为推荐算法的输入源，驱动个性化推荐。

### 5.5 留言反馈

用户可提交留言反馈，支持图片附件。管理员可在后台回复，回复内容在前端展示。管理端首页待办事项中也会显示留言数量。

### 5.6 AI 智能问答

系统集成百度千帆大模型（ernie-4.5-0.3b），提供 AI 智能问答功能：

- 支持文字对话，保留会话历史
- 支持图片、PDF、音频等文件上传分析
- 提供流式（SSE）和非流式两种响应模式
- 登录用户的对话历史会持久化保存到 `chat` 表
- SSE 流式输出：后端通过 `HttpServletResponse.getOutputStream()` 直接写入 `data: content\n\n` 格式的 SSE 事件流，前端通过 `EventSource` 实时接收

### 5.7 WebSocket 私聊

系统基于 WebSocket 实现用户间的实时私聊功能：

- 支持文字、图片、视频、文件和语音消息
- 支持语音录制发送
- 未读消息数量实时提醒（badge 徽标）
- 好友关系通过 `friend` 表管理
- 聊天记录存储在 `chatmessage` 表中

### 5.8 评价反馈

交换完成后，双方可在"评价反馈"模块对交换质量进行评价，包括评价结果和文字反馈。评价内容存储在 `pingjiafankui` 表中。

### 5.9 数据统计（管理端）

管理端首页和统计模块提供基于 ECharts 的可视化数据看板，包括：

- 近30天平台活跃趋势（按发布时间统计）
- 技能分类占比（饼图）
- 注册用户数、技能市场数、交换记录数、学习资料数等概览卡片
- 管理员快速导航（数据统计、用户管理、业务管理、系统配置）

## 六、系统使用指南

### 6.1 用户注册与登录

1. 访问系统首页，点击右上角"登录"
2. 新用户点击"注册"，填写账号、密码、姓名、性别、年龄、手机号等信息
3. 注册成功后即可使用账号密码登录
4. 登录后顶部导航栏显示用户名和个人中心入口

### 6.2 发布技能/需求

1. 登录后进入"技能市场"
2. 点击"发布"按钮
3. 选择类型（"需求"对所有用户开放，"技能"仅限 VIP）
4. 填写标题、技能分类、详细描述、供给图片等信息
5. 提交后等待管理员审核

### 6.3 发起技能交换

1. 在技能列表中浏览或搜索感兴趣的技能
2. 进入技能详情页，点击"申请交换"
3. 填写申请信息并提交
4. 等待对方选择你的技能并确认
5. 收到确认请求后完成确认，交换成功
6. 在"交换记录"中查看所有已完成的交换

### 6.4 购买学习资料

1. 进入"学习资料"页面浏览资料列表
2. 点击资料查看详情
3. 免费资料可直接下载附件
4. 付费资料点击下载后弹出支付窗口
5. 选择支付方式，完成支付后即可下载
6. VIP 用户可直接下载所有付费资料

### 6.5 开通 VIP 会员

1. 进入个人中心的"会员充值"页面
2. 选择适合的 VIP 套餐（月/季/年/永久）
3. 完成支付后 VIP 立即生效
4. 享受发布技能和免费下载资料等特权

### 6.6 AI 智能问答

1. 点击顶部导航栏的"AI智能问答"
2. 在对话框中输入问题
3. 支持上传图片、PDF等文件进行分析
4. 对话历史自动保存，下次登录可继续

### 6.7 管理端操作

管理员登录后台可进行以下操作：

- **业务管理**：技能需求审核与管理、交换申请查看、交换记录查看、评价反馈管理、技能社区管理
- **系统配置**：技能分类、资料类型、资讯分类、论坛类型、公告资讯、学习资料、留言反馈、轮播图
- **用户管理**：用户列表管理、会员充值记录查看
